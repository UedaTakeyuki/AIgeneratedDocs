# 日本語IT技術質問におけるLLMハルシネーションの発生源に関する実証的研究
## 〜コミュニティブログ（Qiita）のデータ占有率と時間的風化が与える影響〜

本リポジトリは、エンピリカルソフトウェア工学（Empirical Software Engineering）のアプローチを用い、**「日本語環境におけるLLM（Gemini, GPT等）のコードハルシネーションの発生源（Root Cause）が、技術コミュニティブログ（Qiita等）にどう依存しているか」**を定量的・客観的に検証する共同研究・議論のためのスペースです。

---

## 💡 本研究の動機・仮説
日本語圏の開発環境は、英語圏に比べて公式ドキュメント（一次情報）の日本語リソースが乏しく、個人ブログやQiita（二次・三次情報）へのデータ依存度が極めて高い現状があります。
LLMは確率統計モデルであるため、以下の構造的課題により、古い記述や動かないコードをサンプリングしやすい仮説を立てています。

1. **情報の時間的風化（Temporal Decay）：** 公式は上書き更新されますが、ブログ記事は「過去のスナップショット」として残り続けます。
2. **サンプリングバイアス：** 被リンク数の強い「古いレガシー記事」がSEOやクローラーに優遇され、AIの出力にノイズとして混入します。

### 📊 フレームワーク別の技術代謝と予測マトリクス
APIの破壊的変更の頻度（API Churn Rate）によって、ハルシネーションに占めるQiitaドメインの寄与率に差が出ると予測し、以下の比較モデルを検証します。

| フレームワーク | カテゴリ | API変化率 | 予測されるQiitaノイズ寄与率（仮説） |
| :--- | :--- | :--- | :--- |
| **Next.js** | フロントエンド | 極高 (App/Pages新旧混在) | **60% 〜 70%** |
| **LangChain** | AI/LLM | 高 (週単位の仕様変更) | **40% 〜 50%** |
| **Flutter** | モバイル | 中 (Null Safety導入など) | **30% 〜 40%** |
| **Ruby on Rails** | バックエンド | 低 (成熟・高い互換性) | **15% 〜 25%** |

## 🛠️ 現在募集しているフィードバック（お気軽にIssueへ！）
本研究をより堅牢なものにするため、エンジニアの皆様からのご意見・コントリビューションを募集しています。

### 1. 「意地悪な質問（テストケース）」の募集
「このフレームワークで、この質問をLLMに投げると、見事に**過去の古いQiita記事の書き方（現在は動かない非推奨コード）でハルシネーションを起こす**」という具体的なプロンプトとAIの回答例があれば、ぜひIssueで教えてください。

### 2. 妥当性への脅威（研究の穴）に関するツッコミ
* 「Zennやはてなブログ、個人ブログとの交絡因子（コピペ・マルチポスト）をどう排除すべきか？」
* 「LLMのGuessing Strategy（当てずっぽうで勝手に嘘をつく確率）と、Qiita由来の嘘をどう区別するか？」
など、リサーチデザインに対する懸念点やアドバイスを歓迎します。

### 3. 検証手法（ロジック）の提案
生成されたコードが「どのQiita記事と字句・構造的に類似しているか」を判定する、AST（抽象構文木）やN-gram、コード埋め込みを用いた効率的な追跡アルゴリズムのアイデア。
